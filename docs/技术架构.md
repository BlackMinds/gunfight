# 技术架构文档

## 1. 架构结论

当前项目是 Nuxt 3 单仓全栈 SPA 游戏：前端页面、Canvas 战斗、配件系统、数值公式、本地存档、Nitro 账号 API 和 PostgreSQL/Neon 云存档都在同一个 Nuxt 项目内，并配置为 `ssr: false` 与 Nitro `preset: 'vercel'`。

2026-07-16 已完成前端模块化、共享领域规则、版本 7 本地存档，以及 Nitro API + JWT + PostgreSQL/Neon 的账号云存档。云存档使用修订号乐观并发控制，客户端发生分叉时必须明确选择本地或云端版本。

下文里的 “Neon” 按你说的 “none” 理解为 Neon Serverless；如果你实际指的是另一个平台，数据库部署段后续替换即可。

## 2. 当前已落地

### 2.1 前端

- Vue 3 + Nuxt 3
- TypeScript
- Pinia 已接入，当前只有轻量 `stores/player.ts`
- `ssr: false`，纯客户端 SPA
- 移动端 H5 viewport 已配置：`width=device-width, initial-scale=1, viewport-fit=cover`
- 全局样式入口：`assets/styles/main.css`
- 主游戏组件：`components/game/GameCanvas.vue`
- 页面入口：`pages/index.vue`、`pages/game.vue`

### 2.2 游戏逻辑

- 当前战斗、角色、敌人、技能、掉落、基地整备、配件背包主要集中在 `GameCanvas.vue`
- 高频战斗对象保留在组件内部状态，不放入 Pinia，避免每帧响应式开销
- 配件模板、配件实例、词条、品质、武器与基础类型在 `shared/game/weapons.ts`
- 关卡、成长、战力等公式在 `shared/game/formulas.ts`
- 当前存档使用 `localStorage` 与云端双写；未登录时保持纯本地模式

当前已识别的架构风险：

- `GameCanvas.vue` 同时承担战斗运行时、基地、背包、结算和大量样式，修改风险过于集中。
- Boss 预览与真实波次各自维护倍率，产生数据漂移。
- 背包容量用数组截断实现，业务损失没有领域事件和界面反馈。
- 本地存档没有明确版本迁移，旧基地强化继续形成隐藏属性。

### 2.3 后端

Nuxt Nitro 能力已经随 Nuxt 可用，且 `nuxt.config.ts` 配置了：

```ts
nitro: {
  preset: 'vercel'
}
```

当前已落地 `server/api/auth/*`、`server/api/cloud-save.*`、JWT 鉴权工具、PostgreSQL schema 和客户端冲突处理。服务端战斗结算属于反作弊与排行榜范围，不影响云存档完整性。

### 2.4 依赖状态

`package.json` 已安装：

- `pg`
- `jsonwebtoken`
- `bcryptjs`
- `@pinia/nuxt`
- `vitest`
- `happy-dom`

当前未安装：

- `@neondatabase/serverless`

如果后续确定数据库用 Neon Serverless，可以选择：

- 继续用 `pg` 连接 Neon PostgreSQL
- 或新增 `@neondatabase/serverless`，为 Vercel Serverless 环境优化连接方式

### 2.5 前端性能预算

- Three.js 角色预览通过异步组件仅在基地装备区按需加载，不进入首屏同步执行链；延迟块当前为 637.19 kB、gzip 164.86 kB。
- 单张构建图片不超过 400 kB；训练区背景 381.17 kB、战斗背景 324.58 kB，配件与装备图集分别为 180.91 kB、161.37 kB。
- 原始 PNG 保留为美术源文件，运行时引用 WebP；本轮 9 张运行时图片由约 14.01 MB 降至约 1.36 MB。
- 生产构建不得出现大于既定延迟块预算的 Vite chunk 告警；如果 Three.js 块继续增长到 700 kB 以上，必须拆分预览能力或替换为更轻量展示。

## 3. 目标架构

目标仍然建议保持 Nuxt 3 单仓全栈应用：

- 前端：Nuxt 3 SPA，负责游戏表现、交互、战斗过程和即时反馈
- 服务端：Nitro Server API，负责账号、存档、结算校验、排行榜等可信逻辑
- 数据库：PostgreSQL，本地用普通 PG，线上优先 Neon Serverless
- 部署：Vercel Serverless

推荐边界：

- 客户端负责高频战斗表现，例如移动、射击、碰撞、特效
- 服务端负责低频可信结果，例如登录、存档、通关奖励、掉落落库
- 数值公式尽量放在 `shared/game/`，保证前端展示和服务端校验使用同一套规则

## 5. 前端设计

### 5.1 页面职责

| 页面 | 当前状态 | 目标职责 |
| --- | --- | --- |
| `/` | 已有 | 首页、进入游戏、基础说明 |
| `/game` | 已有 | 主游戏界面 |
| `/login` | 未建 | 登录 |
| `/register` | 未建 | 注册 |

### 5.2 状态管理

建议拆分：

| Store | 职责 |
| --- | --- |
| `user` | 登录 token、用户信息 |
| `save` | 服务端存档快照、同步状态 |
| `player` | 玩家长期成长属性 |

高频战斗对象不建议进入 Pinia，例如敌人、子弹、掉落物、命中特效、技能冷却帧状态。这些继续放在游戏组件或专门的游戏运行时模块里。

### 5.3 API 封装

当前 `composables/game/useCloudSave.ts` 负责：

- 自动读取用户 token
- 注入 `Authorization: Bearer <token>`
- 统一处理 401、503 和网络错误
- 统一返回业务错误信息

## 6. 后端 API 设计

### 6.1 鉴权

已实现接口：

| 接口 | 作用 |
| --- | --- |
| `POST /api/auth/register` | 注册账号 |
| `POST /api/auth/login` | 登录并返回 JWT |

实现方向：

- `bcryptjs` 负责密码哈希与校验
- `jsonwebtoken` 负责签发 JWT
- `server/middleware/auth.ts` 拦截需要登录的 API
- `/api/auth/*` 和 `/api/health` 放行

### 6.2 存档

已实现接口：

| 接口 | 作用 |
| --- | --- |
| `GET /api/cloud-save` | 拉取玩家存档和云端修订号 |
| `PUT /api/cloud-save` | 按基础修订号保存进度，冲突时返回云端版本 |

当前保存版本化完整快照 JSON，单份上限 1 MB，并通过修订号阻止并发覆盖。

### 6.3 关卡结算

目标接口：

`POST /api/stage/complete`

客户端提交：

- 关卡编号
- 战斗时长
- 击杀数
- 是否击败 Boss
- 战斗摘要

服务端计算：

- 金币奖励
- 经验奖励
- 零件奖励
- 配件掉落
- 是否允许推进关卡

原则：不要接受客户端直接提交的奖励数量。

### 6.4 配件操作

目标接口：

| 接口 | 作用 |
| --- | --- |
| `POST /api/attachment/equip` | 装备配件 |
| `POST /api/attachment/enhance` | 强化配件 |
| `POST /api/attachment/reroll` | 重铸词条 |

当前这些逻辑都在前端内存和 `localStorage` 中，服务端化后需要把消耗、结果和实例编号写入数据库。

## 7. 数据库设计

第一阶段推荐少表 + JSON 存档：

```sql
create table users (
  id uuid primary key default gen_random_uuid(),
  email text not null unique,
  nickname text not null,
  password_hash text not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table player_saves (
  user_id uuid primary key references users(id) on delete cascade,
  save_data jsonb not null,
  save_version integer not null default 1,
  updated_at timestamptz not null default now()
);

create table stage_records (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references users(id) on delete cascade,
  stage integer not null,
  duration_ms integer not null,
  kills integer not null,
  boss_killed boolean not null default false,
  reward_data jsonb not null default '{}'::jsonb,
  created_at timestamptz not null default now()
);
```

第二阶段再把高价值数据拆表：

- `weapons`
- `attachments`
- `equipped_attachments`
- `currencies`
- `leaderboard_records`

这样可以先快速上线云存档，再逐步支持排行榜、交易限制、异常检测和后台运营。

## 8. Vercel Serverless 与 Neon

### 8.1 部署形态

- Nuxt 构建命令：`npm run build`
- Nitro preset：`vercel`
- 前端 SPA 由 Vercel 静态资源承载
- `/api/*` 由 Vercel Serverless Functions 承载

### 8.2 环境变量

建议准备：

```txt
DATABASE_URL=postgres://...
JWT_SECRET=...
BCRYPT_ROUNDS=10
NODE_ENV=production
```

后续如果加入外部定时任务：

```txt
CRON_SECRET=...
```

### 8.3 Serverless 注意事项

- 不依赖服务端内存保存玩家状态
- 不在函数里跑长时间战斗循环
- 单次 API 尽量控制在 60 秒以内
- 数据库连接要适配 Serverless 冷启动和并发
- 高频战斗仍在客户端执行，服务端只接收结算摘要并校验

### 8.4 数据库驱动选择

当前项目已有 `pg`，可以先使用 `pg` 连接 PostgreSQL/Neon。

如果线上连接数或冷启动成为问题，再新增 `@neondatabase/serverless`，并通过 `server/db/client.ts` 隔离实现：

```ts
export async function query<T>(sql: string, params: unknown[] = []): Promise<T[]> {
  // 统一封装数据库查询，避免业务代码关心底层驱动。
}
```

## 9. 暂不需要的模块

以下能力不是当前版本必需：

- Cloudflare R2：目前没有头像、上传图片、玩家自定义资源等需求。
- GitHub Actions cron：目前没有离线收益发放、赛季结算、每日邮件等定时任务。
- 独立 `server/engine/`：当前战斗主要是客户端即时表现，暂时不需要完整服务端战斗引擎。
- 多战斗引擎拆分：当前还没有 PVP、组队、副本、爬塔等多模式复杂度。

等游戏进入账号存档、排行榜、赛季或活动阶段，再补这些模块更合适。

## 12. 与参考架构的差异

当前项目已经收敛为单仓全栈形态，并保留高频战斗在客户端、可信低频数据在服务端的边界。

当前差异：

- 没有 `server/engine/`，暂不做多套服务端战斗引擎
- 已具备账号与云存档 `server/api/`
- 没有 Neon 专用包，当前只有 `pg`
- 没有 R2，暂不需要文件存储
- 没有 cron，暂不需要外部定时任务
- 游戏数据已经有一部分在 `shared/game/`，比放在纯前端镜像更适合未来前后端共用

最终方向可以收敛为：Nuxt 3 单仓全栈应用，前端 SPA 负责游戏体验，Nitro API 负责可信数据，Vercel Serverless + Neon/PostgreSQL 负责线上部署和持久化。
